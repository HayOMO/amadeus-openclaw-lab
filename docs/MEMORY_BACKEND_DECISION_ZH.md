# 记忆方案说明

## 现在的记忆是怎么工作的

Amaduse 的记忆不是把所有聊天一股脑塞进提示词，而是分成三种作用域：

1. 用户记忆：某个成员的称呼、偏好、长期事实和互动印象。
2. 群组记忆：群内共同梗、约定和公共背景。
3. 窗口记忆：某次具体对话的细节，只在需要回忆旧对话时检索。

检索使用“关键词 + 语义向量”的混合方式。名字、外号、原话等精确
信息由关键词检索负责；表达方式不同但意思相近的问题由语义检索负责。
模型只看到命中的少量片段，不会默认读完整聊天历史。

## 为什么这次没有换成 Honcho

Honcho 的优势是能在后台继续推导用户画像和关系信息，适合跨应用、跨会话
的长期个性化。但对现在的 Telegram 群机器人，它有几个尚未解决的问题：

1. 作用域隔离：现有系统明确区分用户、群组和窗口。换后必须证明不会把
   A 群成员的私人记忆带到 B 群，也不能把某个人的印象误当成群体共识。
2. 隐私和可删除性：后台自动推导出的“画像”比用户明确说过的事实更敏感，
   必须能够查看来源、纠错、导出和彻底删除。
3. 额外服务与费用：Honcho 需要单独服务和后台模型调用，会增加网络依赖、
   延迟、故障点和持续费用。
4. 不可控推断：它可能生成听起来合理但并不准确的人物判断。群聊环境里，
   一次玩笑很容易被错误固化成长期事实。
5. 当前性能足够：现有索引为 714 个片段，热态混合检索约 84 毫秒。冷启动
   主要耗时来自首次加载向量模型，之后检索已经足够快。

所以当前决定不是“Honcho 不好”，而是先保留边界更清楚、数据完全本地、
行为容易审计的方案。等 Honcho 能通过作用域泄漏测试、删除测试、召回质量
对比和延迟/费用预算，再考虑迁移。

## RAM 盘和记忆的关系

持久记忆正文不能只放 RAM 盘，因为 RAM 盘重建后内容会消失。Codex App 的
磁盘读写规避属于桌面应用自身的运维问题，不由 Amaduse 项目接管；本项目只
保证用户记忆、群记忆和会话记录保存在可持久化的 OpenClaw 状态目录。
